# Validation Policy

## 검증 계층

### 구조 검증

- 필수 섹션이 존재하는가?
- 에이전트와 스킬이 구분되어 있는가?
- 각 스킬의 입력과 출력이 있는가?
- 호출 관계가 존재하는가?

### 의미 검증

- 목적과 책임이 일치하는가?
- 범위 밖 업무가 명확한가?
- 책임이 중복되지 않는가?
- 스킬 이름이 실제 책임을 나타내는가?

### 실행 가능성 검증

- 호출 조건이 명확한가?
- 필요한 도구가 정의되어 있는가?
- 선행 조건이 있는가?
- 실패 상태가 상위 에이전트로 전달되는가?

### 안전 검증

- 파괴적 작업이 승인 대상으로 분리되어 있는가?
- 비밀 정보 보호 규칙이 있는가?
- 최소 권한 원칙을 적용하는가?
- 외부 영향이 있는 행동을 자동 실행하지 않는가?

### 규약 검증

- 프로젝트 규약 탐색 절차가 있는가?
- 규약 출처를 기록하는가?
- 우선순위가 있는가?
- 충돌 처리 방식이 있는가?

### 사용자 경험 검증

- 모든 누락 정보를 질문하지 않는가?
- 이미 제공된 정보를 다시 묻지 않는가?
- 질문 이유를 설명하는가?
- 가정과 기본값을 표시하는가?

---

## 필수 테스트 유형

- 정상 요청
- 매우 짧은 요청
- 정보 부족
- 규약 제공
- 필수 규약 접근 불가
- 요구사항 충돌
- 고위험 권한 요청
- 범위 밖 책임
- 하위 스킬 실패
- 도구 사용 불가
- 부분 성공

---

## 품질 점수

```yaml
quality_score:
  purpose_clarity: 0-2
  responsibility_boundary: 0-2
  skill_cohesion: 0-2
  input_output_clarity: 0-2
  rule_resolution: 0-2
  permission_safety: 0-2
  question_policy: 0-2
  failure_handling: 0-2
  validation_coverage: 0-2
  maintainability: 0-2
```

- 18~20: 완성도 높음
- 15~17: 사용 가능, 일부 보완 필요
- 11~14: 구조 수정 필요
- 10 이하: 재설계 필요

안전 또는 권한 항목이 0점이면 총점과 관계없이 완료하지 않는다.

---

## 완료 기준

- 목적이 한 문장으로 설명된다.
- 최종 책임자가 명확하다.
- 범위와 범위 밖 업무가 구분된다.
- 하위 스킬이 적절한 크기다.
- 호출 조건, 입력과 출력이 명확하다.
- 규약 탐색과 우선순위가 정의된다.
- 도구와 권한이 최소화된다.
- 승인 대상과 금지 행동이 정의된다.
- 질문 기준이 명확하다.
- 실패와 부분 성공이 정의된다.
- 검증 시나리오가 있다.
- 가정과 미확정 사항이 표시된다.
